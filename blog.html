UX & Visual Design Principles

Short, actionable principles that improve product clarity, trust and usability. Apply these immediately:

1. Start with clear intent (the 3-second test)

Users should understand the page’s primary purpose within ~3 seconds. If they can't, simplify: reduce copy, strengthen headings, and increase visual hierarchy.

2. Use visual hierarchy to guide attention

Size, weight, color and spacing direct attention. Bold primary actions, use color sparingly for calls-to-action, and increase spacing around important elements.

3. Favor progressive disclosure

Show only what's necessary up front. Reveal details on demand (tooltips, accordions, modals) — reduces cognitive load and improves conversion.

4. Microcopy matters

Use concise, benefit-led labels for buttons and inputs. Replace “Submit” with “Request quote” or “Save changes” — clarity beats cleverness.

5. Accessibility is non-negotiable

Contrast (WCAG AA), keyboard navigation, and semantic HTML are essentials. Make error states descriptive and actionable.

6. Motion as a hint, not a show

Use motion to clarify relationships (e.g., where content moved from) and avoid decorative motion that distracts. Respect reduced-motion preferences.

Design Systems: When & How to Build

A design system should be built to solve a real pain — like inconsistent UI, slow engineering handoff, or duplicated visual assets. Here’s a minimal plan:

Why build one?

Gives consistency, speeds up product delivery, and reduces maintenance overhead. But it's not mandatory for very small single-product teams.

Scope & first 3-month checklist

  • Month 0: Audit UI components and tokens (colors, type, spacing).
  • Month 1: Create token file + base components (buttons, inputs, cards) in Figma.
  • Month 2: Ship a lightweight component library (React/Vue) for the most common components.
  • Month 3: Add documentation site + usage examples and establish PR/review process for design changes.

Governance (keep it lean)

A small cross-functional group (1 designer, 1 dev, a PM) should own the system. Prefer documented decisions and a lightweight changelog over heavy approval processes.

When not to build

If product scope is tiny or the team is very early-stage, start with conventions in Figma and iterate only after you have repeatable patterns to justify engineering work.

Figma Tokens — Practical Guide

Design tokens are named values (colors, spacing, type sizes) that keep design consistent and easily transferrable to code. Here’s how to structure them.

Core token categories

  • Color tokens: `color.primary.500`, `color.neutral.100`, `color.error.400`
  • Type tokens: `type.h1.size`, `type.body.lineHeight`
  • Spacing tokens: `space.1`, `space.2`, `space.8`
  • Radius / shadow tokens: `radius.sm`, `shadow.level-1`

A small example token JSON (copy/paste)

{
  "color": {
    "primary": { "50":"#ecfdf5","500":"#10b981","700":"#047857" },
    "neutral": { "50":"#f9fafb","500":"#6b7280","800":"#1f2937" },
    "error": { "500":"#ef4444" }
  },
  "type": {
    "h1": { "size":"32px","lineHeight":"40px" },
    "body": { "size":"16px","lineHeight":"24px" }
  },
  "space": { "1":"4px","2":"8px","3":"12px","4":"16px","8":"32px" },
  "radius": { "sm":"6px","md":"10px" }
}

Use this as a starting point for Figma tokens plugins or to export into your frontend token system.

Naming & versioning

Prefer semantic names (primary / success / error) over ad-hoc color names. Keep tokens in a versioned source (Git) and publish releases when tokens change in breaking ways.
